# cmdproxy

Policy proxy for CLI invocations used by AI agents and shell hooks.

`cmdproxy` evaluates commands in two phases:

1. `rewrite`: normalize command shape into a safer, more canonical form
2. `permission`: decide whether the final command should be `deny`, `ask`, or `allow`

It also lets you test both phases end-to-end from a single config file.

## Value Proposition

`cmdproxy` is not just a dangerous-command blocker.

It exists to make commands run in approved ways and then evaluate permissions on
the resulting final command shape.

Typical examples:

- rewrite `aws --profile prod s3 ls` into `AWS_PROFILE=prod aws s3 ls`
- rewrite `bash -c 'git status'` into `git status`
- rewrite `/bin/ls -R foo` into `ls -R foo`
- deny or ask for commands only after the rewrite pipeline has finished

## Current Status

Today, the codebase supports:

- top-level `rewrite`, `permission`, and `test`
- ordered rewrite steps with optional `match`
- permission buckets `deny`, `ask`, `allow`
- permission evaluation order `deny -> ask -> allow`
- rewrite-local tests
- permission-rule-local tests
- top-level E2E tests
- verified hook artifacts generated by `cmdproxy verify`
- `cmdproxy check`, `cmdproxy doctor`, `cmdproxy verify`,
  `cmdproxy version`, and `cmdproxy hook claude`

## Typical Workflow

1. Initialize user config

```sh
cmdproxy init
```

2. Edit `~/.config/cmdproxy/cmdproxy.yml`

3. Verify and compile the config

```sh
cmdproxy verify
cmdproxy check aws --profile read-only-profile s3 ls
```

4. Register `cmdproxy hook claude --rtk` in Claude Code

## Claude Code Setup

Add a `PreToolUse` Bash hook that calls `cmdproxy hook claude --rtk`.

```json
{
  "matcher": "Bash",
  "hooks": [
    { "type": "command", "command": "cmdproxy hook claude --rtk" }
  ]
}
```

`cmdproxy` evaluates its own policy and then combines that result with Claude
Code settings during a migration/coexistence period. The final result returned
to Claude is:

- `allow`: auto-allow
- `ask`: let Claude prompt the user
- `deny`: block the command

If `--rtk` is enabled, `cmdproxy` evaluates its own permission pipeline first
and then applies the final `rtk` rewrite before returning `updatedInput`.

### Permission Merge Rule

`cmdproxy` and Claude settings are merged with a safety-first rule:

- if either side returns `deny`, the final result is `deny`
- else if either side returns `allow`, the final result is `allow`
- otherwise the final result is `ask`

The important combinations are:

| `cmdproxy` | Claude settings | Final |
|---|---|---|
| `deny` | `deny` | `deny` |
| `deny` | `allow` | `deny` |
| `deny` | `ask` / default | `deny` |
| `ask` | `deny` | `deny` |
| `ask` | `allow` | `allow` |
| `ask` | `ask` / default | `ask` |
| `allow` | `deny` | `deny` |
| `allow` | `allow` | `allow` |
| `allow` | `ask` / default | `allow` |

## Current Config Shape

The currently implemented config file looks like this:

```yaml
rewrite:
  - match:
      command: aws
      args_contains:
        - "--profile"
    move_flag_to_env:
      flag: "--profile"
      env: "AWS_PROFILE"
    strict: true
    continue: true
    test:
      - in: "aws --profile read-only-profile sts get-caller-identity"
        out: "AWS_PROFILE=read-only-profile aws sts get-caller-identity"
      - pass: "AWS_PROFILE=read-only-profile aws sts get-caller-identity"

  - match:
      command_is_absolute_path: true
    strip_command_path: true
    continue: true
    test:
      - in: "/bin/ls -R foo"
        out: "ls -R foo"
      - pass: "ls -R foo"

permission:
  deny:
    - match:
        env_requires:
          - "AWS_PROFILE"
        args_contains:
          - "--delete"
      message: "delete is blocked"
      test:
        deny:
          - "AWS_PROFILE=read-only-profile aws s3 rm s3://example --delete"
        pass:
          - "AWS_PROFILE=read-only-profile aws sts get-caller-identity"

  ask:
    - match:
        command: aws
        subcommand: s3
      message: "s3 operations require confirmation"
      test:
        ask:
          - "AWS_PROFILE=read-only-profile aws s3 ls"
        pass:
          - "AWS_PROFILE=read-only-profile aws sts get-caller-identity"

  allow:
    - match:
        command: aws
        subcommand: sts
        env_requires:
          - "AWS_PROFILE"
      test:
        allow:
          - "AWS_PROFILE=read-only-profile aws sts get-caller-identity"
        pass:
          - "AWS_PROFILE=read-only-profile aws s3 ls"

test:
  - in: "aws --profile read-only-profile sts get-caller-identity"
    rewritten: "AWS_PROFILE=read-only-profile aws sts get-caller-identity"
    decision: allow
  - in: "aws --profile read-only-profile s3 ls"
    rewritten: "AWS_PROFILE=read-only-profile aws s3 ls"
    decision: ask
```

## Design Direction

The current model is pipeline-driven:

- `rewrite`: ordered processing steps that may change command shape
- `permission`: ordered effect buckets evaluated as `deny -> ask -> allow`
- `test`: end-to-end assertions on rewrite plus permission behavior

Caller input stays intentionally simple: a raw command string in, structured
policy evaluation inside `cmdproxy`.
